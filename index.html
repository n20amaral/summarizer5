Version Control Systems (VCS)

What is it?

It's a system that tracks file changes and manages collaborative work on a project.

Why should we use it?

With a VCS it's possible to rollback to a previous version of a file and it's easier to solve conflicts when 2 or more people are working in the same document.
Brevly explain what is a VCS

No Version Control

What's wrong with not using version control?

It's not collaborative work-friendly. Hard to track changes on files. Keeping every version of a file will consume much more storage space.
You will end with something like this picture

What kind of VCS system exists?

Local Data Model Client-Server Model Distributed Model
Refer the other names used in the slides

Local Data Model VCS(aka File Locks)

Require that everyone use the same filesystem In a shared envinroment the file will be locked when someone is using it, making the others wait to finish. Merging local copies of a file nees to be done manually.
Examples:RCS, SCCS
reinforce the struggle

Client-Server VCS (aka Centralized VCS)

Require a connection to the central repository Needs to solve merge conflicts before commit a file. Offline and remote development it's hard to manage
Examples: Subversion, CVS, TFS (proprietary)
Talk about personal experience with subversion

Distributed VCS

  • Does not require a central repository
  • Commits are made to local repository and can be pushed/pulles to/from a remote repository
  • Can be connected to multiple remote repositories.
  • Offline and remote development friendly
Examples: Git, Mercurial, VSTS (proprietary)
note that VSTS uses git also
